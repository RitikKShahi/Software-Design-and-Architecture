Software Reverse Engineering

Extracting design, understanding systems, and recovering lost knowledge.



Reverse engineering can extract design information from source code, but the abstraction level, documentation completeness, and human-tool synergy vary highly.

Objectives of Reverse Engineering

  • Reducing Costs: Finding replacements or cost-effective alternatives for systems/components.
  • Analysis of Security: Examining exploits, vulnerabilities, and malware in cybersecurity.
  • Integration & Customization: Modifying hardware/software components into pre-existing systems.
  • Recovering Lost Source Code: Rebuilding representation when original source is lost or inaccessible.
  • Fixing bugs & maintenance: Repairing flaws in inadequately documented legacy systems.

Reverse Engineering Goals

  • Cope with Complexity: Analyze complex systems to reveal architecture, relationships, and design patterns.
  • Recover lost information: Rebuild source code, analyze data structures, and retrieve design details.
  • Detect side effects: Find unintended implications, dependencies, and interactions not obvious from docs.
  • Synthesis higher abstraction: Abstract low-level features to build higher-level representations for easier analysis.
  • Facilitate Reuse: Find reusable parts or modules to extract and repurpose in other projects.

Reverse Engineering to Understand Data

Often the first reengineering task, occurring at different levels of abstraction.

1. Internal Data Structures (Program Level)

Focuses on defining classes of objects by examining program code and grouping related variables.

Record structures, files, lists, and other data organizations provide initial indicators of abstract data types (classes).

2. Database Structures (System Level)

Global data structures (files, databases) are reengineered to accommodate new DB management paradigms (e.g., flat file to RDBMS or Object-Oriented).

Database Schema Reengineering

Reengineering one database schema into another requires understanding existing objects and relationships.


Steps to Define Data Model:

  • Build an initial object model.
  • Determine candidate keys: Attributes used to point to another record/table become candidate keys.
  • Refine the tentative classes.
  • Define generalizations.

Reverse Engineering to Understand Processing

Begins with extracting procedural abstractions represented by the source code at varying levels:

  • System Level
  • Program Level
  • Component Level
  • Pattern Level
  • Statement Level

Note: For large systems, this is generally accomplished using a semiautomated approach. Output is passed to forward engineering tools.

Reverse Engineering Tools

Manual reverse engineering consumes too much time and labor, hence automated tools are critical.

  • CIAO and CIA: Graphical navigator for software/web repositories.
  • Rigi: Visual software understanding tool.
  • Bunch: Software clustering and modularization tool.
  • GEN++: Application generator for C++ analysis tools.
  • PBS: Software Bookshelf tools for extracting and visualizing program architecture.

Summary


  • Reverse Engineering reconstructs design, architecture, and data structures from existing code.
  • Core Goals: Reduce costs, analyze vulnerabilities, integrate systems, and recover lost knowledge.
  • Data Reengineering: Key focus on recognizing Abstract Data Types (Internal) and Database schemas (Global).
  • Process Flow: Collect -> Examine -> Extract Structure -> Record Logic & Data Flow -> Review -> Document.
